フェーズ 4Security Hub CSPM ✅ 完了（Slack 通知のみ残）操作: 管理アカウント → Aggregator

※「Security Hub CSPM」＝従来のSecurity Hub（本フェーズの対象）。「Security Hub（統合版）」＝2025年登場の相関分析製品（将来検討）。
中央設定とは：Security Hub は本来アカウント×リージョン単位でバラバラに設定するサービス。委任管理者（Aggregator）が「設定ポリシー」を1枚作成して組織全体へ自動配布する仕組みで、新規アカウントにも自動適用され、メンバー側では変更不可（＝統制）になる。

  1. ✅【管理アカウント】委任管理者 = Aggregator を指定。委任後はコンソールが委任先ビューに自動切替。管理 / CloudTrail アカウントで Security Hub コンソールが開けないのは正常
  2. ✅【Aggregator】標準の整理：デフォルト有効だった CIS v1.2.0 を無効化（2018年版・FSBP と重複、コストとノイズ増のみ）。有効な標準 = FSBP v1.0.0 ＋ AWS Resource Tagging Standard v1.0.0（Tagging はタグ未設定リソースを検出。重要度低のため Slack 通知には乗らない）。PCI / NIST は要件なし、CIS が必要になったら最新版（v3.0.0）を選ぶ
  3. ✅【Aggregator】中央設定：ホームリージョン = 東京、リンクリージョン = 大阪、Link future Regions = チェックなし（→ 将来ガバナンスリージョンを追加する際は中央設定へのリージョン追加も忘れずに）、設定ポリシー = FSBP ＋ Tagging、適用先 = 組織全体
  4. ⬜【Aggregator】Slack 通知：findings → EventBridge ルール（フィルタ：Severity = CRITICAL / HIGH ＋ Workflow = NEW ＋ Record State = ACTIVE）→ SNS → AWS Chatbot → Slack。設置は Aggregator × 東京の1箇所（集約先なので組織全体＋GuardDuty 分もカバー）
  5. 【管理アカウント】CT 統合（任意）：SH.xxx コントロールを使う場合のみ Control Tower 側から OU 単位で有効化（Service-Managed Standard が自動作成される）
CT 経由で有効化したコントロールを Security Hub 側から直接無効化しない（ドリフト）。中央設定の開始後は委任管理者を変更できない。

管理アカウントの特別対応（中央設定エラーの解消手順）✅ 解消済み

ポリシーを組織全体に適用すると、管理アカウントだけ「Security Hub is not enabled in the organization management account」エラーで失敗する（委任管理者は管理アカウントに Security Hub を有効化できない仕様）。 かといって管理アカウント側で直接有効化しようとすると「only the administrator can access this operation」でブロックされる（ポリシー関連付けが残っているため）。以下の順で解消する：

  1. 【Aggregator】設定（Configuration）→ 管理アカウントを選択 → 自己管理（Self-managed）に切替（ポリシーの縛りを一時解除）
  2. 【管理アカウント】AWS Config を有効化（Security Hub 有効化の前提条件。東京 → 大阪の順に2回）
    項目東京大阪
    記録戦略 ／ 頻度すべてのリソースタイプ ／ 継続的
    グローバルリソース（IAM 等）の記録ONOFF（両方 ON だと二重記録・二重課金）
    IAM ロールサービスリンクロール（AWSServiceRoleForConfig）
    S3 バケットローカル新規作成（config-bucket-<アカウントID>）東京のバケットを指定（クロスリージョン配信可。履歴ファイルは KB 級のため転送料は月数円未満）
    SNS ／ Config ルールなし（ルールは Security Hub が自動管理）
    組織への集約は S3 の共有ではなく組織 Config アグリゲータ（Aggregator 内）が自動で行うため、レコーダー ON だけで管理アカウントも横断ビューの対象になる。CT の中央バケットへの相乗りは不要
  3. 【管理アカウント】Security Hub を有効化（東京・大阪。標準の選択は何でも可 — 次のステップでポリシーが上書きする）
  4. 【Aggregator】設定 → 管理アカウントを中央設定ポリシーに再関連付け → ステータス「適用済み（Applied）」を確認して完了

  2. 未使用メール＋表示名（命名規則 <project>-<env>-<resource>、例 myapp-dev-app）＋担当者の Identity Center メール＋配置先 OU → 作成（約30分）
  • アカウント作成は必ず Account Factory 経由（Organizations から直接作らない）。ガードレール・CloudTrail・Config 適用済みで払い出される
  • Landing Zone 設定の「アカウントの自動登録」を有効にすると、OU への移動だけで自動登録される（2025年11月〜）
  • 権限セットは AdministratorAccess を安易に配らず用途別に作成。prod への権限付与は最小限のメンバーに
  • Terraform でのアカウント払い出しを標準化したくなったら AFT（Account Factory for Terraform）を検討

追加ガバナンス機能 ⬜ 未着手（いずれも無料）操作: 管理アカウント

① ルートアクセスの集中管理（推奨）

IAM コンソール（管理アカウント）→ アカウント設定 → ルートアクセス管理を有効化（委任管理者：Aggregator）→ メンバーアカウントのルート認証情報を削除。以後ルート作業は短期・タスク限定セッションで中央から払い出す。※管理アカウント自身のルートは対象外（MFA 付きで厳重保管を継続）。

② RCP（Resource Control Policies）

SCP が「人の操作」を制限するのに対し、RCP は「リソース側へのアクセス」を制限（例：組織外から自社 S3 / KMS / Secrets Manager へのアクセスを全面遮断＝データ境界）。Organizations コンソール → ポリシー → RCP を有効化 → Sandbox OU でテストしてから本適用。対応：S3 / STS / KMS / SQS / Secrets Manager。

③ その他（必要になったら）

機能用途導入時期
タグポリシーproject / env タグの強制コスト配分の本格化時
宣言型ポリシーEBS 常時暗号化・AMI 公開禁止などの組織強制ワークロード増加時

LZA（Landing Zone Accelerator）組織設定の IaC 化（CDK ベース・CT の上に被せる公式ソリューション）アカウント数増加・変更履歴管理が必要になったら

  • ガバナンスリージョンは使う分だけ（コストはリージョン数×アカウント数に比例）
  • AWS Budgets で予算アラート設定（例：$20 超過で通知。2個まで無料）
Control Tower の解体（decommission）は手間が大きく、共有アカウント閉鎖に90日待機が必要。検証後も壊さず残して本利用の土台にするのが現実的。

運用の禁止事項（ドリフト・事故防止）

  • 管理アカウントでワークロード（EC2 / S3 等）を動かさない
  • CT が作ったリソース（組織証跡・Config・SCP・共有アカウント内の自動デプロイ資産）を手動変更・削除しない
  • OU・アカウント操作は Control Tower コンソールから行う（Organizations 直接操作は登録漏れの原因）
  • CT 経由で有効化したコントロールを Security Hub 側から無効化しない
  • CloudTrail / Aggregator アカウントを他 OU へ移動しない。Security OU に他のアカウントを入れない
  • アカウント作成は Account Factory 経由。ルートユーザーは緊急時以外使わない
